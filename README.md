# 🔒 AmadeusX

**AmadeusX** — это веб-мессенджер с клиентским сквозным шифрованием (End-to-End Encryption), построенный на Flask, Socket.IO и Web Crypto API. Сервер никогда не видит содержимое сообщений — он хранит только зашифрованный текст, ключи шифрования существуют исключительно в оперативной памяти браузера пользователя.

## ✨ Ключевые возможности 

### Мессенджер
- **Текстовые сообщения** — отправка, получение, шифрование/дешифрование в браузере
- **Голосовые сообщения** — запись через микрофон, шифрование аудио (WebM Opus)
- **Индикатор печатания** — отображается, когда собеседник набирает сообщение
- **Общение в реальном времени** — Socket.IO с WebSocket-транспортом
- **Система приглашений** — для начала общения нужно отправить и принять приглашение
- **Профили пользователей** — аватары, смена пароля, удаление аккаунта
- **Удаление чатов** — полное удаление чата и всей истории

### Безопасность
- **Сквозное шифрование (E2EE)** — AES-256-GCM через Web Crypto API
- **Нулевое знание сервера (Zero-Knowledge)** — сервер хранит только зашифрованные blob-ы
- **Паддинг сообщений** — все текстовые сообщения дополняются до 4 КБ, скрывая реальную длину
- **Огрублённые метки времени** — сервер хранит время с точностью ±5 минут, точное время зашифровано внутри сообщения
- **SRI (Subresource Integrity)** — хеш JavaScript-файла проверяется браузером при загрузке
- **Service Worker** — независимая проверка целостности скриптов, защита от подмены кода на сервере
- **Отпечатки ключей** — визуальная верификация ключа через 8 эмодзи (для защиты от MITM)
- **Исчезающие сообщения** — автоматическое самоуничтожение через 30 секунд (удаление из DOM и БД)
- **Авто-блокировка сессии** — ключ удаляется из памяти после 3 минут бездействия
- **Усиленный PBKDF2** — 600,000 итераций для защиты ключей шифрования от брутфорса
- **Защита от SQL Wildcard Injection** — экранирование спецсимволов `%` и `_` при поиске
- **Timing-safe аутентификация** — защита от перебора пользователей (User Enumeration) через анализ времени ответа
- **CSRF-защита** — Flask-WTF CSRFProtect на всех формах и API
- **Rate Limiting** — ограничение HTTP (200/день, 50/час) и Socket.IO событий (защита от DoS)
- **Content Security Policy (CSP)** — строгая политика с nonce для inline-скриптов
- **Безопасные куки** — HttpOnly, SameSite=Lax, Secure
- **Хеширование паролей** — Werkzeug (PBKDF2 + SHA-256)
- **Валидация входных данных** — жесткая проверка форматов, длины полей и загружаемых файлов (MIME-типы)
- **Безопасный DOM** — отсутствие использования `innerHTML` для предотвращения XSS

### Уровни защиты

| Угроза | Защита |
|--------|--------|
| Перехват трафика (Wi-Fi, провайдер) | HTTPS/WSS + E2EE (AES-256-GCM) |
| Взлом/изъятие базы данных | Zero-Knowledge — только зашифрованные blob-ы |
| Анализ длины сообщений | Паддинг до 4 КБ |
| Анализ времени отправки | Огрублённые метки (±5 мин) |
| Подмена JavaScript на сервере | SRI + Service Worker с проверкой хеша |
| Атака MITM на ключ | Визуальные отпечатки (8 эмодзи) |
| Физический доступ к устройству | Авто-блокировка через 3 мин + исчезающие сообщения |
| CSRF-атаки | CSRFProtect + токены |
| Brute-force | Rate Limiting (Flask-Limiter) |
| XSS-инъекции | CSP с nonce + отключён unsafe-inline |
| Загрузка вредоносных файлов | MIME-валидация + ограничение размера (2 МБ) |

## 📊 Модели данных

### User
| Поле | Тип | Описание |
|------|-----|----------|
| id | Integer (PK) | Уникальный идентификатор |
| username | String(80) | Уникальное имя пользователя |
| email | String(120) | Email (уникальный) |
| password_hash | String(255) | Хеш пароля (PBKDF2 + SHA-256) |
| avatar_url | String(255) | Путь к аватару (nullable) |

### Invitation
| Поле | Тип | Описание |
|------|-----|----------|
| id | Integer (PK) | Уникальный идентификатор |
| sender_id | FK → User | Кто отправил приглашение |
| receiver_id | FK → User | Кому отправлено |
| status | String(20) | `pending` / `accepted` / `rejected` |
| timestamp | DateTime | Время создания |

### Chat
| Поле | Тип | Описание |
|------|-----|----------|
| id | Integer (PK) | Уникальный идентификатор |
| user1_id | FK → User | Первый участник |
| user2_id | FK → User | Второй участник |
| is_active | Boolean | Активен ли чат |

### Message
| Поле | Тип | Описание |
|------|-----|----------|
| id | Integer (PK) | Уникальный идентификатор |
| chat_id | FK → Chat | Привязка к чату |
| sender_id | FK → User | Отправитель |
| content | Text | **Зашифрованный** контент (base64) |
| message_type | String(10) | `text` или `audio` |
| timestamp | DateTime | Огрублённое время (±5 мин) |

> ⚠️ **Важно:** Поле `content` содержит строку формата `salt:iv:ciphertext` (base64). Расшифровать её можно только зная секретный ключ, который вводят пользователи. Точное время отправки зашифровано внутри `content`.

## 🚀 Установка и запуск

### Предварительные требования
- Python 3.10+
- PostgreSQL 14+
- pip

### 1. Клонирование и установка зависимостей

```bash
cd SecurChat/app
python -m venv venv

# Windows
venv\Scripts\activate

# Linux / macOS
source venv/bin/activate

pip install -r requirements.txt
```

### 2. Настройка переменных окружения

Создайте файл `.env` в папке `app/`:

```env
# Flask
SECRET_KEY=ваш_секретный_ключ_здесь
FLASK_DEBUG=False

# PostgreSQL
DATABASE_URI=postgresql://пользователь:пароль@localhost:5432/имя_базы

# CORS (для локальной разработки)
CORS_ORIGIN=http://127.0.0.1:8000

# Security (для production установить в True)
COOKIE_SECURE=False
FORCE_HTTPS=False
```

Сгенерировать `SECRET_KEY`:
```bash
python -c "import secrets; print(secrets.token_hex(32))"
```

### 3. Создание базы данных

```bash
# В PostgreSQL:
CREATE DATABASE имя_базы;
```

При первом запуске таблицы создадутся автоматически (`db.create_all()`).

### 4. Запуск

```bash
cd app
python app.py
```

Приложение будет доступно по адресу: **http://127.0.0.1:8000**

### Для продакшена
1. Поменяйте `CORS_ORIGIN` в `.env` на ваш домен (например, `https://example.com`)
2. Установите `SESSION_COOKIE_SECURE = True` в `app.py`
3. Установите `session_cookie_secure=True` в Talisman
4. Установите `force_https=True` в Talisman
5. Смените `connect-src` на `wss://` в CSP
6. Поменяйте `secure: false` на `secure: true` в Socket.IO клиенте (`scripts.js`)
7. Используйте Nginx/Apache как reverse proxy с SSL (Let's Encrypt)

## 📖 Теория: автоматическая самозащита сервера при компрометации

> Этот раздел носит исключительно образовательный характер. Описанные механизмы используются в сфере кибербезопасности и DevOps для защиты конфиденциальных систем.

### Проблема

При модели E2EE уже отправленные сообщения защищены криптографически. Однако если злоумышленник (или спецслужбы) захватывает контроль над сервером, он может:
1. **Подменить JavaScript-код** — внедрить «закладку», которая при следующем визите пользователя перехватит его ключ шифрования и отправит злоумышленнику.
2. **Собирать метаданные** — кто, когда и как часто общается (даже без расшифровки содержимого).
3. **Перехватывать будущие ключи** — новые пользователи или те, кто зашёл с нового устройства, получат «отравленный» код.

Задача: сделать так, чтобы сервер **автоматически обнаружил вторжение** и самоуничтожился или заблокировался до того, как злоумышленник успеет нанести вред.

---

### Метод 1: Кнопка мертвеца (Dead Man's Switch)

#### Принцип
Сервер ожидает регулярный «пинг жизни» от администратора. Если пинг не поступил вовремя (администратор арестован, потерял доступ, или сервер изолирован), запускается протокол самоуничтожения.

#### Реализация

```
┌──────────────────────────────────────────┐
│           Администратор                   │
│                                          │
│  Каждые 12 часов отправляет:             │
│  POST /deadman/ping                      │
│  Authorization: HMAC-SHA256(timestamp,   │
│                              secret_key) │
└─────────────────┬────────────────────────┘
                  │
                  ▼
┌──────────────────────────────────────────┐
│  Сервер: deadman_watchdog.py             │
│                                          │
│  1. Получил пинг → сбросить таймер       │
│  2. Таймер истёк (12ч + буфер 1ч) →      │
│     ├── DROP ALL TABLES (PostgreSQL)     │
│     ├── shred -vfz -n 5 /app/.env        │
│     ├── shred -vfz -n 5 /app/app.py      │
│     ├── systemctl stop nginx              │
│     └── shutdown -h now                   │
│                                          │
│  Работает как отдельный systemd-сервис   │
│  (не зависит от Flask)                   │
└──────────────────────────────────────────┘
```

#### Псевдокод (Python)

```python
# deadman_watchdog.py — отдельный демон
import time, os, subprocess, hmac, hashlib
from datetime import datetime, timedelta

DEADLINE_HOURS = 13  # 12 часов + 1 час буфер
LAST_PING_FILE = "/var/deadman/.last_ping"
DEADMAN_SECRET = os.environ.get("DEADMAN_SECRET")

def verify_ping(timestamp, signature):
    """Проверка криптографической подписи пинга."""
    expected = hmac.new(
        DEADMAN_SECRET.encode(), timestamp.encode(), hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(expected, signature)

def self_destruct():
    """Полное уничтожение данных."""
    # 1. Удалить все таблицы в PostgreSQL
    subprocess.run(["psql", "-c", "DROP SCHEMA public CASCADE;"])
    # 2. Многократно перезаписать критические файлы
    for f in ["/app/.env", "/app/app.py", "/app/models.py"]:
        subprocess.run(["shred", "-vfz", "-n", "5", f])
    # 3. Остановить веб-сервер
    subprocess.run(["systemctl", "stop", "nginx"])
    subprocess.run(["systemctl", "stop", "securchat"])
    # 4. Выключить машину
    subprocess.run(["shutdown", "-h", "now"])

def main_loop():
    while True:
        # Проверяем, когда был последний пинг
        if os.path.exists(LAST_PING_FILE):
            last_ping = datetime.fromtimestamp(
                os.path.getmtime(LAST_PING_FILE)
            )
        else:
            last_ping = datetime.min

        if datetime.now() - last_ping > timedelta(hours=DEADLINE_HOURS):
            self_destruct()
            break

        time.sleep(60)  # Проверка каждую минуту

if __name__ == "__main__":
    main_loop()
```

#### Защита от ложных срабатываний
- Установить буфер (1–2 часа) сверх основного периода
- Использовать несколько каналов для пинга (HTTP, SMS API, Telegram Bot)
- Перед уничтожением отправить уведомление администратору (опционально)

### Метод 2: Honeytokens (Файлы-ловушки)

#### Принцип
На сервере размещаются «приманки» — файлы с привлекательными именами, которые при нормальной работе никогда не читаются. Если злоумышленник обращается к такому файлу (при ручной разведке или автоматическом копировании), срабатывает тревога.

#### Реализация

```
┌────────────────────────────────────────────────┐
│  Файловая система сервера                       │
│                                                │
│  /app/                                         │
│  ├── app.py                                    │

│  └── .ssh/id_rsa             ◄── ЛОВУШКА 🪤    │
│                                                │
│  Мониторинг: inotifywait / auditd              │
│  При чтении любого файла-ловушки:              │
│  ──► alert + self_destruct()                    │
└────────────────────────────────────────────────┘
```

#### Команда для мониторинга (Linux)

```bash
# honeytrap_monitor.sh
#!/bin/bash
TRAP_FILES=(
  "/app/admin_backup_keys.txt"
  "/app/database_dump.sql"
  "/app/.ssh/id_rsa"
)

for file in "${TRAP_FILES[@]}"; do
  # Создаём файл-приманку с правдоподобным содержимым
  echo "FAKE_DATA_$(openssl rand -hex 32)" > "$file"
done

# Мониторим доступ к файлам через inotifywait
inotifywait -m -e access "${TRAP_FILES[@]}" | while read -r line; do
  echo "[ALERT] Honeytoken accessed: $line"
  # Запуск самоуничтожения
  /app/self_destruct.sh
done
```

### Метод 3: Мониторинг целостности файлов (File Integrity Monitoring)

#### Принцип
Фоновый процесс регулярно вычисляет хеш-суммы всех критических файлов проекта. Если хоть один файл изменён (злоумышленник модифицировал код), а администратор не производил обновления — сервер блокируется.

#### Реализация

```python
# integrity_monitor.py — запускается как отдельный процесс
import hashlib, os, time, json, subprocess

WATCHED_FILES = [
    "/app/app.py",
    "/app/models.py",
    "/app/static/scripts/scripts.js",
    "/app/static/sw.js",
    "/app/templates/base.html",
    "/app/templates/chat.html",
]

HASHES_FILE = "/var/integrity/.file_hashes.json"
CHECK_INTERVAL = 10  # секунд

def compute_hash(filepath):
    sha256 = hashlib.sha256()
    with open(filepath, 'rb') as f:
        for chunk in iter(lambda: f.read(4096), b''):
            sha256.update(chunk)
    return sha256.hexdigest()

def save_baseline():
    """Сохранить эталонные хеши (при деплое)."""
    hashes = {}
    for f in WATCHED_FILES:
        if os.path.exists(f):
            hashes[f] = compute_hash(f)
    with open(HASHES_FILE, 'w') as fp:
        json.dump(hashes, fp)

def check_integrity():
    """Проверить текущие хеши против эталонных."""
    with open(HASHES_FILE) as fp:
        baseline = json.load(fp)

    for filepath, expected_hash in baseline.items():
        if not os.path.exists(filepath):
            return False, f"Файл удалён: {filepath}"
        current_hash = compute_hash(filepath)
        if current_hash != expected_hash:
            return False, f"Файл изменён: {filepath}"
    return True, "OK"

def main_loop():
    while True:
        ok, message = check_integrity()
        if not ok:
            # ТРЕВОГА: файлы были изменены
            # 1. Остановить веб-сервер
            subprocess.run(["systemctl", "stop", "securchat"])
            # 2. Уведомить администратора
            # send_telegram_alert(message)
            # 3. Опционально: уничтожить данные
            break
        time.sleep(CHECK_INTERVAL)
```

#### Важные нюансы
- Эталонные хеши должны храниться в read-only разделе или на отдельном сервере
- Сам скрипт мониторинга тоже должен быть защищён от подмены
- При легитимном обновлении кода нужно обновить эталон (через специальную процедуру с аутентификацией)

### Метод 4: Тревожный пароль (Duress Password)

#### Принцип
Администратор имеет два пароля для входа в панель управления:
- **Настоящий пароль** — обычный вход
- **Тревожный пароль** — внешне работает нормально, но в фоне запускает самоуничтожение

Применяется в ситуации, когда администратора физически заставляют войти в систему.

#### Логика

```
Ввод пароля ──► "correcthorse"  ──► Обычный вход ✅
             │
             └► "correcthorse!" ──► Вход + в фоне:
                                    ├── DELETE FROM message
                                    ├── Уведомление другим админам
                                    └── Через 5 мин: DROP ALL TABLES
```

#### Пример (Flask)

```python
DURESS_PASSWORD_HASH = generate_password_hash("тревожный_пароль")

@app.route('/admin_login', methods=['POST'])
def admin_login():
    password = request.form['password']

    # Проверка тревожного пароля
    if check_password_hash(DURESS_PASSWORD_HASH, password):
        # Внешне — нормальный вход
        login_user(admin_user)
        # В фоне — тихое уничтожение
        threading.Thread(target=silent_wipe, daemon=True).start()
        return redirect('/admin')

    # Обычный вход
    if check_password_hash(REAL_PASSWORD_HASH, password):
        login_user(admin_user)
        return redirect('/admin')

    return "Неверный пароль", 401

def silent_wipe():
    """Тихое удаление всех данных с задержкой."""
    time.sleep(300)  # 5 минут — чтобы не вызвать подозрений
    db.session.execute(text("DELETE FROM message"))
    db.session.execute(text("DELETE FROM chat"))
    db.session.commit()
    # Через 10 минут — полное уничтожение
    time.sleep(300)
    db.session.execute(text("DROP SCHEMA public CASCADE"))
    db.session.commit()
    os._exit(0)
```

### Метод 5: Warrant Canary (Канарейка)

#### Принцип
Это не автоматический, а юридический механизм. Администратор регулярно публикует публичное криптографически подписанное заявление:

> «По состоянию на 08.03.2026, мы не получали запросов от государственных органов на раскрытие данных пользователей. Наш сервер не был скомпрометирован.»

Если заявление пропадает или перестаёт обновляться — пользователи понимают, что что-то произошло.

#### Как это работает

```
┌─────────────────────────────────────────────────┐
│  Раз в неделю администратор публикует на сайте: │
│                                                 │
│  -----BEGIN PGP SIGNED MESSAGE-----             │
│  Дата: 2026-03-08                               │
│  Серийный номер: #0042                          │
│  Заявление: SecurChat не получал                │
│  запросов от спецслужб. Все системы             │
│  функционируют штатно.                          │
│  -----END PGP SIGNED MESSAGE-----              │
│                                                 │
│  Если подпись пропала → сервер скомпрометирован │
│  Если дата устарела → администратор недоступен  │
└─────────────────────────────────────────────────┘
```

#### Почему это работает
Во многих юрисдикциях спецслужбы могут запретить администратору *говорить* о полученном запросе (gag order). Но они не могут заставить его *продолжать публиковать ложное заявление*. Отсутствие нового заявления — это пассивный сигнал, который не нарушает запрет.

### Метод 6: Шифрование диска с ключом в RAM (LUKS + tmpfs)

#### Принцип
Весь диск сервера зашифрован (LUKS). Ключ дешифрования хранится только в оперативной памяти (RAM). При отключении питания или перезагрузке — ключ исчезает, диск становится нечитаемым.

#### Сценарий
1. Спецслужбы приезжают в дата-центр
2. Выдёргивают сервер из розетки (стандартная процедура изъятия)
3. Оперативная память теряет данные через ~2-5 секунд
4. Ключ шифрования утерян навсегда
5. Диск — бессмысленный набор случайных байтов

```bash
# Настройка LUKS + ключ в RAM
cryptsetup luksFormat /dev/sda2
cryptsetup luksOpen /dev/sda2 encrypted_data

# Ключ хранится только в tmpfs (RAM)
mkdir -p /run/keys
mount -t tmpfs tmpfs /run/keys -o size=1M,mode=700
dd if=/dev/urandom of=/run/keys/disk.key bs=64 count=1
cryptsetup luksAddKey /dev/sda2 /run/keys/disk.key
```
